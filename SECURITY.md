# Security Policy

## 🛡️ Our Commitment to Security
We operate under a **Blameless Security Culture**, where we prioritize identifying and fixing system vulnerabilities over individual blame. We appreciate the efforts of security researchers who help us maintain a robust and secure ecosystem.

---

## Supported Versions
We actively provide security updates for the following versions of the project:

| Version | Supported          |
| ------- | ------------------ |
| v1.2.x  | ✅ Yes             |
| v1.1.x  | ⚠️ Critical Only   |
| < v1.0  | ❌ No              |

---

## Reporting a Vulnerability

**Please do NOT open a public GitHub Issue for a security vulnerability.**

To report a vulnerability, please use one of the following private channels:

1.  **GitHub Private Vulnerability Reporting:** Navigate to the "Security" tab of this repository and select "Advisories" to submit a private report. (Recommended)
2.  **Email:** Send a detailed report to **[INSERT SECURITY CONTACT EMAIL]**.

### What to include in your report:
*   A description of the vulnerability and its potential impact.
*   Step-by-step instructions to reproduce the issue (PoC).
*   Any suggested mitigations or fixes.

---

## Our Response Process
Upon receiving a report, the Maintainers will:

1.  **Acknowledge:** Confirm receipt of the report within 48 hours.
2.  **Analyze:** Perform a blameless technical analysis to verify the flaw.
3.  **Fix:** Develop and test a fix. We aim to resolve critical issues within [e.g., 7 days].
4.  **Disclose:** Once the fix is deployed, we will publish a **Security Advisory** to notify users, giving credit to the researcher if desired.

---

## ⚖️ Rules of Engagement
*   Give us a reasonable amount of time to fix the issue before sharing it publicly.
*   Avoid privacy violations, destruction of data, or interruption of our service.
*   Do not perform "Denial of Service" (DoS) attacks or social engineering against our team.

**Thank you for helping us keep this project secure!**
